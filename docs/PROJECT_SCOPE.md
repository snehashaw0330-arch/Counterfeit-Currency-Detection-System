## 1. Vision

A user photographs an Indian banknote with a phone. The system returns a
single, defensible verdict — **REAL / SUSPICIOUS / FAKE** — backed by a
panel of independent, explainable forensic checks and **multiple machine
learning techniques** whose agreement (or disagreement) is shown honestly.

"Great enough" for this project means:

1. **Multiple ML techniques, compared.** Not one model — a benchmarked
   comparison (CNN vs classical models) with a real metrics table. This is
   the literal requirement in the title and in the shared thesis.
2. **Every security feature in the brief is a real, working check** —
   measured from the image, returning PASS/FAIL/INFO with the numbers shown,
   never a hardcoded placeholder.
3. **Honest about limits.** A check that cannot be measured returns INFO,
   not a fake PASS or FAIL. We never claim UV detection we cannot do.
4. **Reproducible.** Anyone can run the test suite + diagnostic harness and
   get the same numbers we report. The benchmark is regenerated by a script,
   not typed by hand.
5. **Explainable.** The frontend shows *why* — per-feature evidence,
   measured aspect vs canonical, digit-size growth, model agreement.

| **Various ML techniques** | MobileNetV2 today (one technique). **+ SVM / RandomForest / KNN / LogReg comparison + benchmark table + fusion** | one → many | **D** |
| Security thread | `security_thread_detection` (morphological, windowed-thread aware) | DONE | shipped |
| Watermark | `watermark_detection` (panel variance) | DONE | shipped |
| Denomination | `denomination_classification` (EasyOCR + palette tiebreak) | DONE | shipped |
| Note proportions | `proportion_analysis` (quad aspect vs canonical RBI mm) | DONE | C-1 |

Every brief line is covered. The two genuinely new build areas are **Phase D**

## 3. Current architecture (what already works — do not rebuild)

```
                       ┌─────────────────────────────────────────┐
  phone photo ──POST──▶│  FastAPI  /predict   (backend/main.py)   │
                       └───────────────┬─────────────────────────┘
                                       │ decode → BGR + 224² tensor
                  ┌────────────────────┴───────────────────────┐
                  ▼                                             ▼
        MobileNetV2 (.keras)                     run_forensic_pipeline()
        REAL/FAKE + confidence                   (backend/forensic.py)
                  │                                             │
                  │                   ┌─────────────────────────┴───────────┐
                  │                   │ _locate_note (auto-crop, Phase B)    │
                  │                   │ 1  structural_sanity   (hard gate)   │
                  │                   │ 2  uv_light_detection  (proxy/INFO)  │
                  │                   │ 3  watermark_detection               │
                  │                   │ 4  ocr_serial_number   (EasyOCR)     │
                  │                   │ 5  gandhi_face_analysis              │
                  │                   │ 6  security_thread_detection         │
                  │                   │ 7  serial_typography_analysis        │
                  │                   │ 8  hologram_detection (colour)       │
                  │                   │ 9  denomination_classification       │
                  │                   │ 10 proportion_analysis (post-pass)   │
                  │                   └─────────────────────────┬───────────┘
                  └───────────────► combined verdict ◄──────────┘
                       score = 0.4·model + 0.6·forensic
                       hard gates: structural / colour / proportion
                       → REAL / SUSPICIOUS / FAKE
                                       │
                       ┌───────────────┴───────────────┐
                       ▼                                ▼
              Next.js frontend (frontend/app/page.tsx) renders every check
```

**Stack:** FastAPI + uvicorn · MobileNetV2 (Keras) · OpenCV · EasyOCR
(PyTorch, primary) + Tesseract (fallback) · Next.js + Tailwind.
**Tests:** pytest (~36 tests across `test_forensic`, `test_phase_a_ocr`,
`test_phase_b_locate`, `test_phase_c_proportions`, `test_api`) +
manifest-driven fixtures (`tests/sample_notes/manifest.json`) +
`tests/diagnostic_harness.py` (investigative confusion-matrix script — the
**source of truth for objective numbers**).

**Response contract (do not break without updating the frontend in the same
commit):** `forensic_analysis` carries the 10 keys above + `modular_ai_pipeline`.
Each check is `{status: PASS|FAIL|INFO, details: str, value?: ...}`.

### Known weak points we will fix (not paper over)
- **MobileNetV2 provenance is opaque** — trained on an unknown set, brittle
  out-of-distribution (approves noise/inverted notes at 95%+; that's *why*
  forensic is weighted 0.6). Phase F retrains it on a documented dataset.
- **Only one ML technique** — Phase D.
- **Motifs/micro-lettering/identification-mark/bleed-lines not yet checked** —
  Phase E.
- **README is stale** ("Placeholder" everywhere) — refreshed alongside this scope.

### Phase D — Various ML techniques + benchmark (the title requirement)

**Goal.** Turn "one CNN" into a benchmarked comparison of multiple ML
techniques, fused into the verdict. This is what makes the project title true.

**D.1 — Dataset assembly.**
- Acquire a public Indian-currency real/fake image dataset
  (Kaggle / Mendeley, as cited in the shared papers) and combine with our
  existing fixtures (23 real + 11 fake/synthetic).
- Land it under `dataset/` (git-ignored — large), with a small committed
  `dataset/MANIFEST.md` describing source, licence, counts per denomination,
  and the train/val/test split (70/15/15, stratified by denom + label).
- A committed `scripts/build_dataset.py` reproduces the split deterministically.

**D.2 — Feature-vector extractor.**
- New module `backend/features.py`: `extract_feature_vector(bgr) -> np.ndarray`
  that turns the forensic signals into a fixed numeric vector:
  serial-typography metrics (slope, R², growth), proportion deviation, palette
  hue-entropy + mean-saturation, security-thread component count, watermark
  variance, structural edge/std stats, **and an LBP histogram** (the technique
  the shared thesis used) over canonical note regions.
- Pure function, unit-tested, deterministic. Reused by both training and
  inference so there is no train/serve skew.

**D.3 — Train the classical models.**
- `scripts/train_classical.py`: train SVM, RandomForest, KNN, LogisticRegression
  (+ GradientBoosting if it helps) on the D.2 vectors. 5-fold CV, fixed seed.
- Persist each fitted model + the scaler to `models/classical/` (git-ignored)
  and a small committed `models/classical/metrics.json`.

**D.4 — Benchmark report (the deliverable table).**
- `scripts/benchmark_models.py` emits `docs/BENCHMARK.md`: accuracy, precision,
  recall, F1, and a confusion matrix per technique (CNN + each classical model),
  evaluated on the held-out test split. Regenerated by script, never hand-typed.

**D.5 — Fusion into the verdict.**
- Load the **best** classical model as a second opinion in `backend/main.py`.
- Fuse by a **calibrated rule-based** layer (extend the existing
  `0.4·model + 0.6·forensic`) — e.g. `model_cnn`, `model_classical`, `forensic`
  with documented weights chosen against the diagnostic harness. **No learned
  meta-classifier yet** (overfits on small data). The learned-stack upgrade is
  Phase D-stretch, gated on dataset size ≥ a documented threshold.
- Response gains `ml_models: {cnn, classical, agreement}` without removing any
  existing key.

**D.6 — Surface it.**
- Frontend "Model Comparison" panel: each technique's verdict + confidence and
  an agreement indicator. Read-only; existing layout untouched elsewhere.

**Acceptance (Phase D):**
- `docs/BENCHMARK.md` regenerates from script and shows ≥ 4 techniques with a
  full metrics table + confusion matrices.
- Best classical model ≥ 0.80 F1 on the held-out test split (or we document
  honestly why the data ceiling is lower).
- All existing tests pass; new `tests/test_phase_d_features.py` (feature-vector
  shape/determinism) + `tests/test_phase_d_fusion.py` (fusion never raises,
  response keys intact) pass.
- Diagnostic harness: **no regression** vs current baseline.

### Phase E — Visible-light security-feature checks (figures, motifs, …)

**Goal.** Implement the security features the brief and the shared papers name,
using visible-light image processing only. Each is a real measurement
returning PASS/FAIL/INFO, wired into the pipeline + frontend, with fixtures
and unit tests — same discipline as every existing check.

**E.1 — Motif / figure template matching.** Per-denomination reference crops
(Gandhi portrait, Ashoka pillar emblem, RBI logo, denomination numeral motif);
ORB/feature-match or normalised template correlation in the expected region.
PASS on confident match, FAIL on clear absence on a confidently-denominated
note, INFO otherwise.

**E.2 — Micro-lettering presence.** High-frequency texture-energy / gradient
density in the band where "RBI" + denomination micro-text sits. Real notes
carry fine repeating structure; flat photocopies do not. INFO when resolution
is too low to assess (honest — phone photos often can't resolve micro-text).

**E.3 — Identification mark / intaglio.** The raised-print geometric mark
(for the visually impaired) differs by denomination (circle, square,
triangle, etc.). Detect its shape in the expected location.

**E.4 — Bleed lines.** Count the angular bleed lines on the left/right edges
(present on ₹100/200/500/2000, count varies by denomination). Compare against
the expected count for the OCR'd denomination.

**E.5 — See-through register.** Front/back numeral alignment. Single-side
input limits this — document the constraint and return INFO when only one
side is available; PASS/FAIL only when both halves are assessable.

**Acceptance (Phase E):**
- Each new check has: a PASS, a FAIL, and an INFO unit test;
  ≥ 1 manifest fixture exercising it; wired into `run_forensic_pipeline`,
  `EXPECTED_KEYS`, the frontend grid, and the TypeScript `ForensicAnalysis` type.
- Diagnostic harness: real bucket no worse, new fakes caught where designed.
- No existing test regresses. New keys added; no existing key removed.

### Phase F — Model retraining + verdict calibration (production ML)

**Goal.** Replace the opaque MobileNetV2 with one trained on the documented
Phase D dataset, and calibrate the fused verdict thresholds against evidence.

- F.1 Retrain/fine-tune the CNN on the D.1 dataset with augmentation; proper
  train/val/test; per-denomination eval. Save with a `models/MODEL_CARD.md`
  (data, metrics, intended use, limitations).
- F.2 Select verdict thresholds from ROC/PR curves on the harness corpus —
  replace the hand-set `0.65 / 0.35` with documented, evidence-chosen values.
- F.3 Publish honest accuracy **per grade** (clean scan / phone / fake) in
  `docs/BENCHMARK.md`.

**Acceptance (Phase F):** CNN test-set metrics in the model card; harness
verdict counts improve or hold vs Phase D; thresholds traceable to a curve.

## 5. Non-goals and honest limitations

- **No UV/IR hardware detection.** UV stays a visible-light proxy, INFO-only,
  never a verdict driver. Stated plainly in the UI and report.
- **No cloud deployment.** Local demonstration only.
- **No multi-currency.** Indian Rupees only.
- **No 100% accuracy claim.** Physically impossible from a phone photo without
  sensors. Honest target: solid, defensible accuracy per grade, reported with
  its confusion matrix — not a marketing number.
- **See-through register is single-side-limited** and returns INFO when only
  one side is uploaded.

## 6. Working agreement

- **One phase at a time, in order.** No starting the next until the current
  phase's acceptance criteria hold.
- **No placeholders.** Every check measures something real or returns an
  honest INFO. No hardcoded PASS/FAIL.
- **Solid fixes, not patches.** If a check is wrong, fix the method — don't
  special-case the failing fixture.
- **The harness is the source of truth.** Anything the user reports as "broken
  on this photo" becomes a named fixture under `tests/sample_notes/` so we
  never regress on it.
- **Contract stability.** Never change a response shape, the frontend
  `ForensicAnalysis` type, or `EXPECTED_KEYS` without updating all three in the
  same commit.
- **Reproducibility.** Benchmarks and dataset splits come from committed
  scripts with fixed seeds, never hand-edited numbers.
- **Session handoff.** At the end of every working session, update
  [STATUS.md](STATUS.md) so the next chat resumes instantly.

### Definition of done (per phase)
1. Acceptance criteria in this doc are met and shown (test output / harness
   numbers pasted into the commit or STATUS.md).
2. New unit tests cover the happy path **and** the INFO/failure path.
3. Frontend + types + `EXPECTED_KEYS` updated if the contract changed.
4. STATUS.md and (if needed) README/SETUP updated.
5. A single focused commit.

### Risk register
| Risk | Mitigation |
|---|---|
| Public dataset too small/noisy → weak ML numbers | Report honestly per grade; rule-based fusion (not learned stack); document ceiling |
| EasyOCR cold-start latency | Already pre-warmed at startup; keep it |
| Template matching brittle across note series | Per-series references + INFO fallback; never FAIL on low confidence |
| Contract drift breaks frontend | Same-commit rule + shape tests in `test_forensic.py` |
| Scope creep into UV/hardware | Explicit non-goal; UV frozen as proxy |
